# Access Token vs Refresh Token

## Access Token

* Short-lived token (example: 15 minutes)
* Stored in React memory/state
* Sent in API requests as:

```txt id="t06yze"
Authorization: Bearer <token>
```

Used for accessing protected APIs.

---

## Refresh Token

* Long-lived token (example: 7 days)
* Stored in HTTP-only cookies
* Sent from backend using:

```js id="4tk2ln"
res.cookie("refreshToken", refreshToken)
```

and then:

```js id="j0czr8"
res.json({ accessToken })
```

Frontend JavaScript cannot directly access HTTP-only cookies, making them more secure.

---

# Refresh Token Flow

When the access token expires:

```txt id="1c1x5y"
401 Unauthorized
```

Frontend automatically calls:

```txt id="nuznfd"
POST /api/users/refresh
```

Backend:

* verifies refresh token
* extracts user information
* generates a new access token for that user

Logout should clear the refresh token cookie.

# JWT Token Architecture: Session vs User Authentication

## Current Implementation (Phase 1)
The JWT token generated is a **session-specific deep-link token** with these characteristics:
- **Payload**: `{ sessionId: "<mongo_id>", type: "personalization" }`
- **Purpose**: One-time session access via email link
- **Expiry**: 7 days (TTL in MongoDB)
- **No user identity**: Doesn't contain userId, email, or roles

## Future User Authentication JWT
When adding user login/registration, create a **separate JWT**:
- **Payload**: `{ userId, email, roles, iat, exp }`
- **Purpose**: User authentication across the app
- **Expiry**: Configurable (e.g., 1 day with refresh tokens)
- **Contains**: User identity and permissions

## Why Separate?
Different purposes require different tokens. Session JWT is for passwordless, one-time access. User JWT is for authenticated users with persistent access. Mixing them creates security risks and couples unrelated concerns.
